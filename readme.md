# LightRPC
A Light and Efficient RDMA-based RPC Framework

### 引言

------

事实上，基于RDMA构建RPC在学术界有很多优秀的工作，例如：Herd（SIGCOMM14），RFP（EuroSys17），ScaleRPC（EuroSys19），eRPC（NSDI19），Flock（SOSP21），HatRPC（SC21）等等。这些工作都具有一定的创新性，也都在解决他们所提出的难题。但是其中许多工作也隐藏着许多的“问题”（注：除了Anuj Kalia大佬，其他基本都没有开源，具体的实现细节以及实际效果无法得知），会让人觉得：这个设定或者这个方法真的能用到生产中吗？举两个简单的例子：

* RFP的前半部分我觉得特别好，但是关于客户端如何去获取RPC响应的讨论，我不是很认同。对于服务端，入站的性能好于出站，所以客户端read好于服务端write（观点：在某些场景下，确实是这样）。由于客户端不知道何时read响应，所以引入了一个启发式的算法（观点：如果客户端的CPU资源可以任意浪费，这样确实也可以，但是仍然有个问题是：这个响应占用的内存不需要释放吗？服务端怎么知道客户端read完毕呢？）。
* ScaleRPC用分组管理以及虚拟映射的方法提升RDMA的扩展性，令人耳目一新。我们知道，RDMA在RC模式下难以扩展的原因是，RNIC的片上cache是有限的，当连接、内存注册等过多时，缓存争用和缺失会导致性能的下降。以往，大家的研究重点是：在大规模的场景下，如何尽可能减少需要缓存的内容（例如共享QP、使用大页内存等等）。但是，cache容量终究是有限的，挤牙膏似的“优化”也有可能会导致“丢了西瓜、捡了芝麻”。ScaleRPC则是将优化的粒度提升到了一个个的客户端，比如通过对客户端进行分组并赋予一定的时间片来提高cache的命中率。但是ScaleRPC使用RDMA的方式是这样的：服务端将一大块内存分成若个zone（每个zone对应一个客户端），每个zone又分为若干block（每个block对应一个RPC调用），每个zone会绑定一个轮询线程，该线程通过忙轮询的方式判断是否有请求到达。这里存在两个问题：一是block的大小决定了消息的大小，所以不具有通用性。二是一个线程对应一个客户端，并且还是忙轮询的方式，服务端的CPU是肯定吃不消的。

因此，做LightRPC的想法是：作为学院派，能不能在考虑性能的同时，兼顾生产场景中的需求呢？在提升通用性的同时，如何优化RDMA的使用流程呢？由于能力以及时间原因，LightRPC同样只是个toy，但是也包含了自己独有的思考。

进度：目前，我们完成了单个客户端（单线程以及多线程）与服务端交互的性能测试。下面，我们会进行：（1）多节点下一个服务端与多个客户端（非对称场景）的测试；（2）多节点且每个节点既存在服务端又存在客户端（对称场景）的测试。（3）将框架嵌入到其他系统或者实际应用中的测试。
